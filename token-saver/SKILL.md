# Token Saver

## 核心原则

上下文窗口是公共资源。此技能的意图是让 Codex 在保持有效性的前提下，尽可能减少
每一次交互的 token 消耗。采用以下策略时，保持质量与输出完整性的平衡。

## 响应压缩策略

### 1. 精简回复格式

- 使用短段落，避免冗长铺垫。直接给出结论或答案，再视需要补充理由。
- 不使用问候语、结束语、过渡填充句（"当然"、"没问题"、"让我来"等）。
- 不使用 emoji、em dash、感叹号等非必要符号。
- 列表项 ≤ 3 时使用行内逗号分隔，≤ 6 时使用扁平列表，超过 6 项才考虑嵌套。
- 代码变更优先使用 `apply_patch` 做精确编辑，而非重写整个文件。
- 引用文件时使用绝对路径的 clickable markdown 链接，避免重复写出文件内容。

### 2. 减少文件加载

- 优先用 `rg` / `Select-String` 搜索特定内容，而非读取整个文件。
- 并行读取：将多个文件的读取操作合并到 `multi_tool_use.parallel` 中。
- 不重复加载已在上下文中存在的文件内容。
- 对于大文件，先读取头部（`head -n 100`）或搜索关键行，而非完整加载。
- 不使用 `cat` 或其他 shell 写文件的方式；用 `apply_patch` 做精确编辑。

### 3. 编码效率

- 一次性代码中可使用短变量名（如 `i`, `n`, `s`, `arr`），但保持可读性。
- 省略显而易见的注释；仅在算法复杂或设计决策非明显时添加简短注释。
- 合并循环和操作以减少代码行数。
- 优先使用标准库/既有工具链，避免引入新的依赖。
- 使用现成的脚本（`scripts/`）替代每次重复编写相同逻辑。

### 4. 上下文管理

- 使用渐进式披露：只在需要时加载引用文件和详细文档。
- 子任务完成后立即清理子 agent，不保留不必要的上下文。
- 当技能有多个变体时，只在 SKILL.md 保留核心工作流，变体细节放在 `references/`。
- 避免加载 `references/` 目录中与当前任务无关的文件。
- 在长对话中，主动建议进行上下文压缩/重置以释放空间。

### 5. 工具选择

- 确定性操作优先用 `exec_command` 执行 shell 命令，而非在 context 中编写代码。
- 批量文件操作使用 `multi_tool_use.parallel` 并行化。
- 文件编辑使用 `apply_patch` 精确修改，避免全文重写。
- 搜索使用 `rg` / `Select-String` 而非读取 + 手动扫描。
- `git show` 比 `cat` 更高效地查看文件历史版本。

## 默认触发行为

当此技能被触发时，Codex 应自动切换到"节省 token 模式"：

1. 所有回复默认采用最紧凑的表达方式。
2. 除非用户明确要求详细说明，否则省略步骤说明、理由推演和可选方案对比。
3. 优先选择 token 消耗更低的工具调用路径。
4. 在 commentary 更新中保持同样紧凑的风格。

## 边界情况

- 如果用户明确要求详细解释/完整输出，优先满足用户需求而非节省 token。
- 如果节省 token 导致输出不可用或有 bug，优先保证正确性。
- 此技能不应影响代码质量、功能完整性和安全性。
